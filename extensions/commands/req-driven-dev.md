# Requirement-Driven Development

Transform user requirements into verified implementation through systematic workflow with progressive disclosure documentation structure.

## Key Features

- **Progressive Disclosure**: Requirements split into focused sub-documents for AI readability
- **Context-Based Generation**: Generate requirements from session/planned tasks/code when directory empty
- **Smart Supplement Logic**: Update existing related requirements vs creating duplicates
- **Index & Tracking**: Master index + active tracker for easy navigation
- **Archive System**: Completed requirements archived with traceable naming

## Usage

```bash
# Basic usage with requirement text
/req-driven-dev "Add user authentication with OAuth2 support"

# Specify output directory
/req-driven-dev "Implement payment integration" ./docs/requirements

# Specify both output and historical directories
/req-driven-dev "Add notification system" ./docs/requirements ./docs/legacy-requirements

# With context file reference
/req-driven-dev "Refactor API layer based on architecture-review.md"

# Mark requirement as completed (triggers archive)
/req-driven-dev --complete REQ-20260314-143025
```

## Parameters

| Parameter | Default | Description |
|-----------|---------|-------------|
| Requirement | required | The requirement text or description to implement |
| OutputDir | `docs/requirements` | Directory for requirement documents |
| HistoryDir | `docs/requirements` | Directory containing historical requirements for reference |
| `--complete REQ-ID` | optional | Mark requirement as completed and archive |

## Workflow Phases

1. **Discovery** - Analyze context, check existing requirements, determine supplement vs independent
2. **Documentation** - Generate progressive disclosure document structure
3. **Planning** - Create detailed execution plan with user confirmation
4. **Execution** - Implement with continuous verification against requirements
5. **Validation & Archive** - Test, verify, and archive completed requirements

## DOCUMENT STRUCTURE (Progressive Disclosure)

### Directory Layout

```
{OutputDir}/                              # Default: docs/requirements/
├── requirements-index.md                  # Master index of all requirements
├── active-requirements.md                 # Current work tracker
├── req-{timestamp}-{slug}/               # Requirement directory
│   ├── index.md                          # Summary + progress (entry point)
│   ├── functional.md                     # Functional requirements (FR-*)
│   ├── non-functional.md                 # Non-functional requirements (NFR-*)
│   ├── technical.md                      # Technical approach + constraints
│   ├── test-strategy.md                  # Test plan + verification
│   ├── plan.md                           # Execution plan
│   └── verify.md                         # Verification report
└── _archive/                              # Completed requirements
    └── req-{timestamp}-{slug}/
        ├── index.md
        ├── ... (all sub-documents)
        └── _archived-at.md               # Archive metadata
```

### Why Progressive Disclosure?

| Problem | Solution |
|---------|----------|
| Large monolithic docs overwhelm AI | Split into focused sub-documents |
| Context window exhaustion | Load only needed sections |
| Hard to find specific info | Index + sub-doc structure |
| Update conflicts | Granular file updates |

### File Naming Conventions

```
Directory: req-{timestamp}-{slug}/
- timestamp: yyyyMMdd-HHmmss (e.g., 20260314-143025)
- slug: lowercase, hyphen-separated, max 50 chars from title

Examples:
req-20260314-143025-add-dark-mode/
req-20260310-090000-user-authentication/
req-20260312-150530-payment-integration-stripe/
```

## IMPLEMENTATION INSTRUCTIONS

### Phase 1: Discovery

**Step 1.1: Check Requirements Directory State**

Check if OutputDir exists and contains requirement directories:
- If empty or doesn't exist → STATE="empty"
- If contains req-* directories → STATE="populated"

**Step 1.2: Context-Based Generation (When Empty)**

If directory is empty, generate initial requirements from:

1. **Session Context Analysis**:
   - Review current conversation for user requests
   - Extract implicit requirements from discussion
   - Identify patterns in user's questions

2. **Planned Tasks Analysis**:
   - Check for TODOs in code comments (`// TODO:`, `// FIXME:`, `// HACK:`)
   - Review any task tracking files
   - Look for incomplete implementations

3. **Code Analysis**:
   - Identify incomplete implementations
   - Find placeholder code or stubs
   - Detect architectural gaps

**Step 1.3: Supplement vs Independent Decision (When Populated)**

If directory has existing requirements, determine relationship:

```
DECISION LOGIC:
1. Scan all existing requirement index.md files
2. For each existing requirement:
   - Extract: title, affected_components, functional requirements
   - Calculate similarity score (0.0-1.0) based on:
     * Keyword overlap (40%)
     * Component overlap (30%)
     * Semantic similarity (30%)
3. If highest_similarity >= 0.70:
   → SUPPLEMENT existing requirement
4. Else:
   → CREATE independent requirement
```

**Similarity Factors:**

| Factor | Weight | Calculation |
|--------|--------|-------------|
| Keyword Overlap | 40% | Jaccard similarity of key terms |
| Component Overlap | 30% | Intersection / union of affected_components |
| Semantic Similarity | 30% | Related concepts (auth/login, payment/checkout) |

**Step 1.4: Ask Clarifying Questions**

Ask clarifying questions **ONLY** when:
- Multiple valid interpretations exist with significantly different implementations
- Critical technical decisions have no clear best path
- Scope ambiguity could lead to 2x+ effort difference
- Supplement vs independent decision is borderline (0.65-0.75 similarity)

**MUST DO:**
- Parse requirement text to identify core functionality, dependencies, affected components
- Explore codebase for architecture, patterns, file structure
- Reference historical requirements in `HistoryDir`
- Determine supplement vs independent before documentation

**MUST NOT DO:**
- Do NOT start implementation before documentation is complete
- Do NOT assume requirements without evidence
- Do NOT create new requirement without checking similarity
- Do NOT skip exploration of existing patterns

### Phase 2: Documentation

**Step 2.1: Create Directory Structure**

```
For NEW requirement:
{OutputDir}/req-{timestamp}-{slug}/
├── index.md
├── functional.md
├── non-functional.md (if needed)
├── technical.md
└── test-strategy.md

For SUPPLEMENT to existing:
{OutputDir}/req-{existing-timestamp}-{existing-slug}/
├── index.md (UPDATE Progress + Supplement History)
├── functional.md (APPEND new FR-*)
├── non-functional.md (APPEND new NFR-* if needed)
├── technical.md (UPDATE if approach changes)
└── test-strategy.md (UPDATE test plan)
```

**Step 2.2: Generate Documents**

| Document | Required | Generated From |
|----------|----------|----------------|
| index.md | ✅ | Title, summary, context, related requirements |
| functional.md | ✅ | User requirements → FR-* items with acceptance criteria |
| non-functional.md | ⚪ | Performance, security, usability requirements |
| technical.md | ✅ | Architecture analysis, constraints, approach |
| test-strategy.md | ✅ | Functional requirements → test cases |

**Step 2.3: Update Index Files**

1. **Create requirements-index.md** if doesn't exist
2. **Update requirements-index.md**:
   - Add new requirement to "Active Requirements" table
   - If supplementing: update existing row status/phase
   - If new: add row with link, status, priority, summary

3. **Create active-requirements.md** if doesn't exist
4. **Update active-requirements.md**:
   - Add to "Currently In Progress" if starting immediately
   - Add to "Up Next" if pending dependencies
   - Add to "Blocked" if has blocking requirements

**Step 2.4: Update Supplement History (When Supplementing)**

Add entry to `index.md` Supplement History table:
```markdown
| {datetime} | Supplemented | Added FR-003: {description} |
```

**MUST DO:**
- Create directory structure for each independent requirement
- Generate all required sub-documents (index, functional, technical, test-strategy)
- Update requirements-index.md on every new/updated requirement
- Update active-requirements.md on status changes
- Add supplement history entries when updating existing requirements

**MUST NOT DO:**
- Do NOT create single monolithic requirement document
- Do NOT skip index updates after requirement creation/update
- Do NOT exceed 50 characters in slug
- Do NOT delete archived requirements (preserve for reference)

### Phase 3: Planning

**Step 3.1: Generate Execution Plan**

Create `plan.md` inside requirement directory:

1. **Break down into atomic tasks**:
   - Each task has clear input, output, and success criteria
   - Identify dependencies between tasks
   - Mark tasks that can run in parallel

2. **Present plan to user**:
   - Display numbered task list with dependencies
   - Show estimated complexity for each task
   - Highlight any decisions requiring user input

**Step 3.2: User Confirmation**

- Ask user to confirm the plan or request modifications
- Allow user to add/remove/reorder tasks
- Proceed only after explicit approval
- Update `plan.md` approval section on confirmation

**Step 3.3: Update Tracker**

Update `active-requirements.md` with:
- Current phase
- Task progress

**MUST DO:**
- Create plan inside requirement directory (`plan.md`)
- Include test tasks in plan when implementation is involved
- Get explicit user approval before execution

**MUST NOT DO:**
- Do NOT skip user plan approval
- Do NOT omit testing from plan

### Phase 4: Execution

**Step 4.1: Execute Tasks**

1. Execute tasks in dependency order
2. Run parallel tasks simultaneously where possible

**Step 4.2: Track Progress**

After each task:
- Verify against task acceptance criteria
- Update `index.md` Progress table
- Update `active-requirements.md` status
- Update requirements document if discoveries require changes

**Step 4.3: Handle Discrepancies**

- Compare implementation outputs against requirements document
- Add discrepancies to `plan.md` as new tasks if needed
- Document changes in `index.md` Supplement History

**MUST DO:**
- Update `index.md` Progress table after each phase
- Update `active-requirements.md` on status changes
- Document any requirement changes

**MUST NOT DO:**
- Do NOT skip verification after task completion
- Do NOT proceed to next phase if current phase verification fails
- Do NOT modify requirements without documenting the change

### Phase 5: Validation & Archive

**Step 5.1: Run Tests**

Execute all tests defined in `test-strategy.md`:
- Unit tests
- Integration tests
- Manual verification steps

**Step 5.2: Generate Verification Report**

Create `verify.md` with:
- Acceptance criteria results
- Test results
- Issues found (if any)
- Final verdict (PASS/FAIL)

**Step 5.3: Archive Process (When PASS)**

If verification passes:

1. **Create archive directory**:
   ```
   {OutputDir}/_archive/req-{timestamp}-{slug}/
   ```

2. **Move all documents**:
   - Preserve original directory name
   - Move all sub-documents

3. **Create archive metadata**:
   - Create `_archived-at.md` with completion info

4. **Update requirements-index.md**:
   - Remove from "Active Requirements"
   - Add to "Completed (Last 10)" section
   - If > 10 completed, remove oldest from display

5. **Update active-requirements.md**:
   - Remove from "Currently In Progress"
   - Update any dependent requirements status

**Archive Naming Convention:**

```
_archive/
└── req-{original-timestamp}-{original-slug}/
    ├── index.md
    ├── functional.md
    ├── non-functional.md
    ├── technical.md
    ├── test-strategy.md
    ├── plan.md
    ├── verify.md
    └── _archived-at.md    ← Archive metadata added
```

**Why preserve original name?**
- Traceability: Same ID across lifecycle
- Searchability: Find by timestamp or slug
- Consistency: Links remain valid

**MUST DO:**
- Generate verification report before archive
- Move to `_archive/` with original name preserved
- Create `_archived-at.md` metadata
- Update index (move Active → Completed, keep last 10)
- Update tracker (remove from In Progress)

**MUST NOT DO:**
- Do NOT archive requirement with unverified acceptance criteria
- Do NOT delete archived requirements
- Do NOT remove completed requirements from index (keep last 10)
